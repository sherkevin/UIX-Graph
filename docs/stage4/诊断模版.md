# 故障诊断路径统一台账与分角色表单模板

本文档对原来的多张表做一次收敛，目标是同时满足两件事：

- `统一`：最终保留一张全量主台账，汇总专家信息和开发补全信息，方便全局统计、筛选、查询和直接指导开发
- `分离`：另外保留两张分角色表单，一张给专家访谈时使用，一张给开发线下补全时使用

## 1. 收敛后的最终结构

不再并列维护“场景表、节点表、连线表、证据表、技术映射表、Gap 表”六类独立表。  
改为只保留三张表：

1. `全量主台账`
2. `专家访谈表`
3. `开发补全表`

三张表的关系是：

- `专家访谈表` 负责先把业务路径问出来
- `开发补全表` 负责把业务语言翻译成配置、库表和 mock 方案
- `全量主台账` 是最终归档和查询入口，由前两张表按统一主键回填得到

## 2. 设计决策

### 2.1 推荐统一粒度

三张表统一采用“`每个节点一行`”作为基本粒度，而不是“每条边一行”。

这样做的原因：

- 专家访谈时，按“当前看什么、怎么判断、下一步去哪”来聊最自然
- 开发补全时，也更容易围绕一个节点补齐 `metric`、`action`、数据源和验证方案
- 树和回路不再靠单独的连线表维护，而是收敛进节点行中的 `分支定义` 字段

### 2.2 树和回路如何表达

在每个节点的 `分支定义` 字段中，按“每个分支一行”的方式填写：

```text
BR01: 条件=-20 < {output_Mw} < 20; 去向=N022,N023,N024; 类型=parallel
BR02: 条件=else; 去向=N099; 类型=fallback
BR03: 条件={n_88um} < 8; 去向=N010; 类型=loop_back
```

约定：

- `去向` 可以是单个节点，也可以是多个节点
- `类型` 只允许：`single`、`parallel`、`fallback`、`loop_back`
- 一个节点是否形成树、并行或回路，不靠额外表单，而靠这个字段表达

## 3. 三张表的定位

### 3.1 全量主台账

用途：

- 最终归档
- 统计所有场景、节点、根因、指标、数据源、待确认项
- 作为开发、测试、配置评审时的唯一总入口

特点：

- 信息最全
- 同时包含专家口径和开发补全
- 不直接拿给专家填

### 3.2 专家访谈表

用途：

- 访谈专家时使用
- 只保留专家能回答、也应该回答的字段
- 尽量不出现库表、字段、`source_kind` 之类技术词

特点：

- 字段最少
- 只问业务路径、判断口径、页面入口、脚本和经验阈值

### 3.3 开发补全表

用途：

- 访谈结束后，开发把专家语言补成配置和数据实现
- 记录数据库映射、配置字段、mock、待确认项和验证方式

特点：

- 不要求专家参与
- 允许记录“当前还没确认”的技术 gap

## 4. 三张表的统一主键

三张表必须共享以下主键字段：

- `场景编号`
- `节点ID`

推荐命名：

- 场景编号：`SCENE-模块-序号`
- 节点ID：`N001`、`N002`、`N003`

补充字段：

- `父节点ID列表`
- `分支定义`

这样就可以：

- 用 `场景编号 + 节点ID` 把专家表和开发表合并回主台账
- 用 `父节点ID列表 + 分支定义` 表达树结构和回路

## 6. 专家访谈表字段模板

原来的表 6 把“节点本身的信息”和“节点之间的关系/分支信息”混在了一张表里，  
专家在填写时既要描述节点属性，又要同时描述路径跳转，实际很难一次写清。  

因此这里改成 **两张表**，可直接理解为 Excel 的两张 sheet：

1. **节点属性表**：只填写“这个节点本身是什么”
2. **关系表**：只填写“节点和节点之间怎么连、整条 trace 怎么走”

这样拆开后，专家填写逻辑会清晰很多：

- 先把所有节点逐个盘清楚
- 再把这些节点如何串成一条诊断 trace 单独填写

图谱按 **深度优先** 思路串联；父节点状态由 **子节点状态** 的逻辑运算得出。故障树语义下 **不设「正常分支」**，表中也不再保留与之对应的字段。发生故障时从父节点继续向下遍历，被激活的部分图中，除 **数据来源** 外的源头节点即故障根因。

### 6.1 专家访谈表 A：节点属性表

这张表一行只描述 **一个节点** 的属性，不描述跳转关系。


| 字段名             | 说明                                                 |
| --------------- | -------------------------------------------------- |
| 场景编号            | 可先由团队预填；标识当前节点属于哪个故障场景                             |
| 故障现象名称          | 当前讨论的是哪个故障现象                                       |
| 所属模块/分系统        | 专家口径；该节点主要归属哪个模块/分系统                               |
| 适用机台/机型         | 适用范围                                               |
| 节点ID            | 团队预编号；要求全场景唯一                                      |
| 节点类型            | 三选一：`故障现象` / `数据表征` / `数据来源`；仅 **数据来源** 节点需要专家截图佐证 |
| 节点名称            | 这一节点的业务名字                                          |
| 专家判断对象          | 专家这一节点“看什么”                                        |
| 专家获取方式          | 在哪看、怎么拿                                            |
| 专家看到的字段名/文案     | 业务系统里的真实名称                                         |
| 前端路径/功能路径       | 页面入口、菜单、导出路径                                       |
| 是否为数据表征或故障现象根节点 | `是/否`；仅 `故障现象` / `数据表征` 判断，数据来源节点不作为根因源头           |
| 根节点输出根因         | 仅上述根节点填写；如果是源头，输出什么根因                              |
| 根节点输出分系统        | 仅上述根节点填写；如果是源头，输出哪个分系统                             |
| 专家备注            | 其他经验说明                                             |


补充说明：若阈值为独立的 **数据来源** 节点（与指标分开），须在指标节点与阈值节点之上增加 **中间节点** 承载逻辑运算（如 **与运算**），运算含义写在中间节点；这里是建模规则，不作为 Excel 表头填写。

### 6.2 专家访谈表 B：关系表

这张表一行只描述 **一条关系/一条路径片段**，不重复描述节点属性。


| 字段名      | 说明                                                         |
| -------- | ---------------------------------------------------------- |
| 场景编号     | 对应哪个故障场景                                                   |
| trace    | 一条诊断 trace，填写格式固定为 `[node_id1,node_id2,node_id3,node_id4]` |
| trace_id | 本条诊断路径的唯一id，全局唯一                                           |
| trace名称  | 这一trace的业务名字                                               |
| trace目的  | 这一trace在排查什么                                               |


### 6.3 trace对应的edges表

这张 edges 表用来描述节点与节点之间的 **计算关系**，每一张 edges 表对应一个 trace_id。边可理解为寄生在 `father_node` 上的虚拟运算节点：专家访谈时仍按“边”来问，但落配置时把它作为父节点状态的激活规则。


| 字段名         | 说明                                                                                                                                                  |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| trace_id    | 本表对应的trace_id，全局唯一                                                                                                                                  |
| edge_id     | 边详细信息表id，全局唯一                                                                                                                                       |
| son_nodes   | 子节点ID列表；本条边参与计算的子节点集合                                                                                                                               |
| father_node | 父节点ID；本条边寄生在该父节点上，用于计算这个父节点状态是否激活                                                                                                                   |
| action      | **Action 边**：填写字典，函数名为 key；后续在 condition 中以 `{函数名}` 形式引用                                                                                            |
| 是否使用脚本      | `是/否`                                                                                                                                               |
| 脚本名称/文件/负责人 | 若有脚本必须记录                                                                                                                                            |
| condition   | **Condition 边**：填 action 的逻辑运算表达式，必须按 Python 逻辑表达式书写；action 函数名必须用 `{}` 包裹，如 `({has_alarm}) or (({metric_abnormal}) and ({threshold_hit}))`；无则填 `无` |
| description | 虚拟运算节点的文字描述                                                                                                                                         |


### 6.4 关系表填写约定

为了让关系表真的能落到配置，建议在“边属性”这一列里按固定结构填写，至少包含下面这些信息：

- `edge_id`：边编号
- `son_nodes`：参与本次计算的子节点集合
- `father_node`：被计算、被激活的父节点
- `edge_action`：**Action 边** 填字典，函数名为 key，并作为 condition 中可引用的 action
- `edge_condition`：**Condition 边** 填 action 的 Python 逻辑表达式或 `无`，action 函数名必须写成 `{action_func}`，如 `({has_alarm}) or (({metric_abnormal}) and ({threshold_hit}))`
- `loop_back`：是否回到前面，`是/否`
- `remark`：补充说明

如果后续团队决定把“边属性”继续拆细成多列，也建议保持这几个核心字段不变（故障树不设「正常分支」类字段）。整体按状态转移图理解：父节点是否激活，取决于其子节点状态经过本条边定义的逻辑运算后的结果。

填写补充约定：

- 边从属于前面的 `father_node`，`trace_id` 以前面这个 `father_node` 所在 trace 为准。
- `action` 建议按字典填写，例如 `{has_alarm: "是否存在报警", metric_abnormal: "指标是否异常"}`；函数名统一使用小写蛇形命名（`lower_snake_case`），建议采用 `动词_对象` 或 `判断词_对象`，避免中文、空格和临时编号。
- `description` 是该虚拟运算节点的规范化文字描述，建议统一写成“当 <子节点状态/动作条件> 满足 <逻辑表达式> 时，激活 
  ”。

---

1. 指标与阈值拆分不是要填写的字段，而是对6.1的一个补充说明

2. action应该是一个字典，函数名为key

3. 边从属于前面那一个father_node，填写的trace_id以前面那个father_node的trace为准

4. 虚拟节点的description是该虚拟节点的运算的文字描述，要统一化规范化

5. action函数名的命名也要统一化规范化

## 9. 填写流程

1. 团队先预填 `场景编号`、`节点ID`
2. 访谈时只使用 `专家访谈表`
3. 访谈结束后，开发只使用 `开发补全表`
4. 最终把两张表按 `场景编号 + 节点ID` 合并进 `全量主台账`
5. 开发、测试、评审、统计统一只看 `全量主台账`

## 11. 当前项目下的特别提醒

- 当前运行时权威配置仍是 `config/diagnosis.json` 和 `config/reject_errors.diagnosis.json`
- 当前引擎已经支持树状路径、并行分支和回路，不需要再为“回路”额外设计独立表
- 现阶段最关键的 gap 仍然是：
  - 专家知道页面和脚本，不知道库表字段
  - 开发知道库表和配置，不知道专家真实排查路径
